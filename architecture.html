Hivemind-Core Architecture

Privacy-preserving Postgres inside a TEE. Users write raw SQL. Reads go through sandboxed agents. Nobody -- including the operator -- sees individual rows. Only agent-mediated, scope-constrained, mediator-audited answers leave.

2. Query Pipeline (Async, Multi-Stage)

Clients submit agent archives via POST /v1/query-agents/submit. The server returns a run_id immediately. Build + pipeline stages execute in the background. Clients poll GET /v1/query-agents/runs/{run_id} for per-stage progress.

6. Security Layers (Defense in Depth)

graph TB
    L0["Layer 0: Encrypted Storage\nLUKS2 disk + TDX RAM encryption"]
    L1["Layer 1: Simulation + Tape\nScope agent tests query agent, audits LLM calls"]
    L2["Layer 2: Scope Function Firewall\nPost-query row filtering, fail-closed"]
    L3["Layer 3: SQL Validation\nsqlglot AST: parse errors surface, non-SELECT blocked"]
    L4["Layer 4: Budget Enforcement\nHard caps on tokens + LLM calls per query"]
    L5["Layer 5: Mediator\nLLM output audit, PII redaction via response tags, no tool access"]
    L6["Layer 6: Notarized Deploys\nIPFS log + blockchain allowlist before keys released"]

    L0 --- L1 --- L2 --- L3 --- L4 --- L5 --- L6

9. Stage Timing (RunStore)

Every pipeline execution is tracked in _hivemind_query_runs with per-stage start/end timestamps and final output. Clients poll GET /v1/query-agents/runs/{run_id} to see stage progress and retrieve results.

ColumnTypeDescription
run_idTEXT PKUnique run identifier
agent_idTEXTQuery agent that ran
statusTEXTpending / running / completed / failed
build_started_at / build_ended_atFLOATDocker image build
scope_started_at / scope_ended_atFLOATScope agent resolution
query_started_at / query_ended_atFLOATQuery agent execution
mediator_started_at / mediator_ended_atFLOATMediator processing
outputTEXTFinal output (truncated to 10k chars)

10. Information Flow: What Data Reaches the LLM

This traces exactly what information leaves the TEE boundary (to the LLM provider) at each pipeline stage. Green = data entering a stage. Purple = LLM calls. Blue = tool calls. Orange = data leaving a stage. Red = security enforcement points.

Concrete Example Trace

Query: "What are the top 5 most active users?"

CLIENT REQUEST
POST /v1/query-agents/submit {archive: agent.tar.gz}
← {run_id: "abc123", agent_id: "xyz", status: "pending"}

STAGE 0: BUILD (background task)
Extract archive, build Docker image
RunStore: build_started_at, build_ended_at recorded

STAGE 1: SCOPE AGENT (Docker container, FULL_READ access)
Container env vars: QUERY_PROMPT="What are the top 5 most active users?", QUERY_AGENT_ID="default-query"
LLM call #1 (via Bridge → OpenRouter → Claude):
  system: "You are a scope agent. Your job is to write a Python scope function that acts as a query firewall..."
  user: "Determine scope for this query: What are the top 5 most active users?"
Tool calls (via Bridge → Host process):
  get_schema → returns full table/column/type definitions
  execute_sql("SELECT * FROM users LIMIT 5") → returns raw unfiltered rows (scope agent has FULL_READ)
  list_query_agent_files → returns ["agent.py"]
  read_query_agent_file("agent.py") → returns full query agent source code
LLM call #2-N: multi-turn conversation incorporating tool results. Raw DB rows + schema + agent source are sent to the LLM.
  simulate_query(scope_fn_source="def scope(sql, params, rows): ...") → runs nested query agent
Output: {"scope_fn": "def scope(sql, params, rows):\n return {'allow': True, 'rows': [redact(r) for r in rows]}"}
Host validates: AST-check (no imports/exec/eval/dunders) → compile in restricted namespace

STAGE 2: QUERY AGENT (new Docker container, SCOPED access)
Container env vars: QUERY_PROMPT="What are the top 5 most active users?"
LLM call #1 (via Bridge → LLM):
  system: "You are a query agent with access to a database via SQL tools..."
  user: "What are the top 5 most active users?"
Tool calls (via Bridge → Host):
  get_schema → returns schema
  execute_sql("SELECT username, COUNT(*) as activity FROM events GROUP BY username ORDER BY activity DESC LIMIT 5")
    → sqlglot validates: must be SELECT, parse errors surfaced → scope_fn(sql, params, rows) → filtered rows returned to agent
LLM call #2-N: multi-turn with filtered query results. Only scope-approved data reaches the LLM.
Output: "The top 5 most active users are: User A (1,203 events), User B (987 events)..."

STAGE 3: MEDIATOR (new Docker container, NO data access)
Container env vars: RAW_OUTPUT="The top 5 most active users are...", QUERY_PROMPT="What are the top 5 most active users?"
LLM call #1 (via Bridge → LLM):
  system: "PRIVACY GATE. You are a strict output mediator... Wrap final response in <response> tags..."
  user: "QUERY_PROMPT:\nWhat are the top 5 most active users?\n\nRESPONSE TO AUDIT:\nThe top 5 most active users are..."
No tools available.
Post-processing: extract content from <response> tags, strip reasoning preamble
Output: "The top 5 most active accounts had between 650 and 1,200 events."

CLIENT POLLS
GET /v1/query-agents/runs/abc123
{output: "The top 5 most active accounts had between 650 and 1,200 events.", status: "completed", build: 2.1s, scope: 8.3s, query: 12.1s, mediator: 3.4s}
